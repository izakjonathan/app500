V149 Saved Games Persistence Fix

Built from V148 after examining the saved-games logic.

Findings:
- Saved Games were stored as one localStorage array only.
- If that one array failed, was overwritten, or was cleared by browser storage cleanup, the list could disappear.
- Individual games still exist in Supabase by game ID, but the local library/list could be lost.
- URL opening had a bug: when opening ?game=<id> that was not in the local library, the app could fall back to the last locally saved game instead of waiting for the shared URL game.

Fixes:
- Added redundant Saved Games storage:
  - primary library array
  - backup library array
  - game ID index
  - individual per-game backups
- Saved Games library is rebuilt by merging:
  - primary array
  - backup array
  - individual per-game records
  - current legacy game
- Removed the 50-game cap from the library upsert.
- Added placeholder protection so “Loading shared game...” cannot replace an existing real game.
- Fixed URL loading so ?game=<id> does not fall back to a different saved local game.
- Delete now removes the individual per-game backup too.
- Added “Repair list” in Saved Games to rebuild the library from backups.

Important:
- iOS/Safari can still clear browser localStorage under storage pressure.
- Supabase shared game rows remain the real cross-device source when you have the invite link/room ID.
- This version makes the local Saved Games list much harder to lose accidentally.

No scoring/sync/gameplay logic changed.
